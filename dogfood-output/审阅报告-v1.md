## 二、详细问题清单

### field-parser.js（字段解析引擎）

#### F1 [严重] 组合尺寸正则贪婪匹配多点数 — 第186-190行, 第506行
**问题描述:** 正则 `[\d.]+` 会贪婪匹配任意数字和小数点的组合。输入 `"1.2.3×4×5"` 时，第一个捕获组会匹配 `"1.2.3"`，然后 `parseFloat("1.2.3")` 返回 `1.2`（静默截断），导致尺寸数据错误。

**代码:**
```javascript
const combinedDimPattern = /^[\d.\s]+\s*[×xX*]\s*[\d.\s]+\s*[×xX*]\s*[\d.\s]+$/;
const splitMatch = cleaned.match(/^([\d.]+)\s*[×xX*]\s*([\d.]+)\s*[×xX*]\s*([\d.]+)$/);
```

**修复建议:** 将 `[\d.]+` 改为 `[\d]+(?:\.[\d]+)?`（整数或标准小数），确保每个维度只有一个可选小数点。

#### F2 [严重] detectUnitFromHeader 正则无法匹配 "meter" — 第239行
**问题描述:** 正则 `/met[er]{2}/` 中 `[er]{2}` 表示 "e" 或 "r" 字符恰好出现2次，能匹配 "metre"（t-r-e 中 re 匹配），但 "meter" 是 m-e-t-e-r，`[er]{2}` 只能匹配最后的 "er"，前面的 "met" 后面需要紧接匹配。实际上 `"meter".match(/met[er]{2}/)` 返回 `null`，因为 "met" 后面是 "e"，然后 `[er]{2}` 需要再匹配两个字符，但只剩 "r"。

**验证:**
```javascript
"meter".match(/met[er]{2}/)   // null ❌
"metre".match(/met[er]{2}/)   // "metre" ✓
```

**修复建议:** 改为 `/met(?:er|re)s?/` 或 `/meters?/`。

#### F3 [严重] extractItems 中组合尺寸时序问题 — 第496-518行
**问题描述:** 组合尺寸列的数据在第517行被 `normalizeNumber(row[lengthCol])` 处理。如果 `normalizeNumber` 先执行，`*` 号会被 `replace(/[^\d\.\-]/g, '')` 删除，导致后续的正则匹配 `/[×xX*]/` 失败，组合尺寸无法拆分。

**代码路径:**
```javascript
if (lengthCol >= 0) {
    const rawStr = String(row[lengthCol] || '').trim();
    if (/[×xX*]/.test(rawStr)) {  // ← 如果 normalizeNumber 先执行，这里可能失败
        // 拆分逻辑
    }
    if (isNaN(rawL)) rawL = normalizeNumber(row[lengthCol]);  // ← 但这里在 if 之后
}
```

**辩证分析:** 当前代码中 `normalizeNumber` 在 `if` 块之后执行，所以时序上没有问题。但如果未来重构调整顺序，就会触发此bug。建议添加防御性注释或重构为独立的组合尺寸处理函数。

#### F4 [警告] inferUnit 边界值1500被误判为cm — 第210-221行
**问题描述:** `avg > 2000 → mm`，`avg < 3 → m`，否则 `cm`。如果实际数据是1500mm（常见包装尺寸），会被推断为15cm（0.15m），导致货物变成蚂蚁大小。

**修复建议:** 放宽 mm 阈值到 `avg > 500` 或 `max > 1000`，或增加更智能的推断逻辑（结合数量级分布）。

#### F5 [警告] isDataRow 过于激进 — 第260-277行
**问题描述:** `skipWords` 包含 `'total'`，如果型号列包含 `"TOTAL-123"` 或 `"Subtotal"`，整行会被跳过。子串匹配方式过于宽松。

**修复建议:** 将子串匹配改为整词匹配或仅检查非型号列。

#### F6 [警告] 脏数据 "2 . 12" 无法处理 — 第503-505行
**问题描述:** 当前正则处理 `"2 .12"` 和 `"2. 12"`，但 `"2 . 12"`（数字+空格+点+空格+数字）无法被匹配。

**修复建议:** 增强正则：`/(
\d)\s*\.\s*(
\d)/g`。

#### F7 [建议] netWeight 字段冗余 — 第45-47行
**问题描述:** `grossWeight` 已包含所有中文重量关键词（单重、单位重量、净重），`netWeight` 只有英文关键词且永远不会被优先匹配。

**修复建议:** 移除 `netWeight` 或将其关键词合并到 `grossWeight` 并添加注释说明。

### container-db.js（集装箱数据库）

#### C1 [严重] validateItem 死代码 — 第248-267行
**问题描述:** `validateItem` 函数定义完整但全代码库中没有任何调用点。这意味着完整的约束校验管道实际上未被使用，所有校验分散在 `packing-engine.js` 的 `emsPlace` 中单独进行。

**影响:** 如果 `emsPlace` 漏掉某些校验（如单件重量 > payload），`validateItem` 本应作为后备防线，但实际上不存在。

**修复建议:** 在 `packing-engine.js` 的 `packSingleContainer` 或 `emsPlace` 中调用 `validateItem`，或将其逻辑合并到放置流程中。

#### C2 [警告] getOrientations 重复方向 — 第225-237行

**修复建议:** 删除第227行。

#### C3 [警告] checkSizeConstraints 错误信息误导 — 第116-124行

#### C4 [建议] CONTAINER_DB volume 浮点精度 — 第12-13行

### packing-engine.js（装箱核心算法）

#### P1 [严重] normalizeSpaces 过度封锁 — 第151-173行
**问题描述:** 当空间A被空间B包含时，如果A.blockedAbove=true，B.blockedAbove 被设为 true。但B可能还包含其他子空间C，C上方本来可以放货物，现在因为B被封锁，C也无法使用了。

**代码:**
```javascript
if (a.blockedAbove && !b.blockedAbove) {
    b.blockedAbove = true;  // 过度封锁！
}
```

**修复建议:** 不应传播 blockedAbove 到包含空间，而是在 emsPlace 中逐个检查空间的 blockedAbove 属性。

#### P2 [严重] detectOverlaps 缺少 epsilon — 第718-731行
**问题描述:** AABB 重叠检测使用严格比较 `a.x < b.x + b.l`，没有 epsilon。浮点精度误差可能导致：
- 两个刚好接触的货物被误判为重叠（如 `1.0000000001 > 1.0`）
- 或两个实际重叠的货物因精度问题未被检测（较少见）

**修复建议:** 添加 epsilon：`a.x < b.x + b.l - 1e-6`。

#### P3 [警告] emsPlace 死代码（double blockedAbove check）— 第233行, 第251行
**问题描述:** 第233行 `if (space.blockedAbove && space.z > 0.001) continue;` 和第251行 `if (space.z > 0.001 && space.blockedAbove) continue;` 逻辑完全相同。

**修复建议:** 删除第251行并添加注释说明。

#### P4 [警告] dblScore yScore 权重不合理 — 第206-217行
**问题描述:** `zScore * 1000 + xScore * 10 + yScore`。yScore 未乘系数，导致 y 方向优先级与 x 方向几乎同级（xScore*10 范围 0-10，yScore 范围 0-1）。这可能导致货物在宽度方向分布不合理。

**修复建议:** 统一权重：`zScore * 1000 + xScore * 100 + yScore * 10`。

#### P5 [警告] purgeOverlapSpaces epsilon 对小件货物不足 — 第179-195行
**问题描述:** epsilon=0.001m=1mm。如果货物尺寸为 5mm（0.005m），两个货物间隙小于 1mm 时，空间可能被错误剔除。

**修复建议:** 将 epsilon 改为相对值，如 `Math.min(0.001, minItemDim * 0.1)`。

#### P6 [建议] autoRetry 重算不比较结果 — 第737-752行
**问题描述:** recalibrate 换排序策略重算后，不比较新旧结果优劣，直接返回新结果。如果新结果更差（如利用率更低），用户反而吃亏。

**修复建议:** 比较新旧结果，选择更优方案返回。

### app.js（应用控制器）

#### A1 [严重] resetState 未清除 _calculating — 第1553-1572行
**问题描述:** `resetState()` 清除了大部分状态，但遗漏了 `_calculating`。如果用户在计算过程中点击重置，`_calculating` 仍为 true，后续计算按钮将永久不可用（除非刷新页面）。

**修复建议:** 在 resetState 中添加 `state._calculating = false;`。

#### A2 [严重] renderContainerRecommendation 强制重置 useMixed — 第804行
**问题描述:** 每次渲染 Step 3 时，`state.useMixed = false` 强制重置混合装箱选择。如果用户之前启用了混合装箱，返回上一步再回来，选择会丢失。

**修复建议:** 仅在首次渲染时重置，或保留用户选择。

#### A3 [警告] showError 双重 fallback 仍可能静默失败 — 第121-139行
**问题描述:** 如果 targetSelector 和 stepEl 都不存在，alert 不会被添加到 DOM。虽然这种情况极少，但在动态 DOM 操作后可能发生。

**修复建议:** 最终 fallback 到 `document.body.prepend(alert)`。

#### A4 [警告] handleFileUpload JSON.stringify 去重性能 — 第292-298行
**问题描述:** 大数据量时（10000行×100列），`JSON.stringify` 去重可能阻塞主线程数秒。

**修复建议:** 使用更轻量的去重方式，如逐列比较或采样去重。

#### A5 [警告] confirmData volume 字段不一致 — 第659行
**问题描述:** `mergedItems` 中 `volume: selectedGroup.l * selectedGroup.w * selectedGroup.h` 是单件体积，但 quantity 是 totalQty。原始 item 的 volume 也是单件体积，这保持一致。但如果 UI 层期望显示总体积，就会出错。

**辩证分析:** 经检查，UI 层（renderDataPreview 和 renderSkuTable）不显示 volume 字段，所以当前无影响。但数据模型语义模糊。

**修复建议:** 添加注释说明 volume 是单件体积，或改为 totalVolume。

#### A6 [警告] simulateProgress 异常时进度不自动隐藏 — 第192-221行
**问题描述:** 如果 onDone 回调异常被外层 catch，progress 会被隐藏（第1183行）。但如果异常发生在 simulateProgress 内部（如 messages 数组越界），progress 不会隐藏。

**修复建议:** 在 simulateProgress 内部添加 try-finally 确保 progress 隐藏。

#### A7 [建议] getTV 轮询超时过短 — 第1396-1418行
**问题描述:** 20次×500ms=10秒超时。在慢网络或低性能设备上可能不够。

**修复建议:** 增加到 30秒或提供手动刷新按钮。

### three-viewer.js（3D可视化）

#### T1 [严重] containerSpec L/W/H undefined 导致 NaN 几何体 — 第127行
**问题描述:** 虽然第409行检查了 `!container`，但没有检查 `container.L/W/H` 是否为有效数字。如果 `containerSpec` 为 `{}` 或部分缺失，`new THREE.BoxGeometry(NaN, ...)` 会产生无效几何体。

**修复建议:** 添加验证：`if (!container || !isFinite(container.L) || !isFinite(container.W) || !isFinite(container.H))`。

#### T2 [警告] document.hidden 时 OrbitControls 事件积累 — 第340行
**问题描述:** 页面隐藏时跳过渲染，但 OrbitControls 仍在监听鼠标/触摸事件。页面重新可见时，相机会跳变到累积的位置。

**修复建议:** 在 hidden 时也调用 `ctrl.update()` 或暂停事件监听。

#### T3 [建议] disposeAll 未释放 texture — 第55-90行
**问题描述:** 当前代码未使用 texture，所以安全。但如果未来添加货物贴图，会内存泄漏。

**修复建议:** 添加 texture 释放逻辑（防御性编程）。

### pdf-exporter.js（PDF导出）

#### D1 [警告] 长货物清单分页异常 — 第134-150行
**问题描述:** 如果单个集装箱有数百件货物，y 坐标递增 `y += 4` 可能超出 pageH 但未触发分页检查（分页检查只在循环开头）。

**修复建议:** 在每次 `doc.text()` 后检查 y 坐标，必要时 addPage。

### index.html / styles.css

#### H1 [警告] .hidden !important 与 inline style 冲突风险 — styles.css 第703行
**问题描述:** `.hidden { display: none !important }` 优先级过高。虽然 app.js 已改用 `classList.add/remove('hidden')`，但如果第三方库或未来代码使用 `style.display = 'block'`，会被覆盖。

**辩证分析:** 当前代码已修复此问题（使用 classList），但 CSS 规则本身仍是潜在风险。
