# 计划：基于《操作规范》反向审查代码实现与功能流转

## 1. 摘要

基于用户确认的《集装箱装箱模拟系统 — 操作规范与代码映射》文档大纲（混合组织），反向审查当前代码实现与功能流转。目标是：
- 识别代码实现与行业/业务规范不一致之处
- 找出功能流转中的断点、歧义和潜在 bug
- 输出一份可执行的修复与改进计划

## 2. 当前状态分析

### 2.1 系统功能流转（已确认）

```
上传文件 (Step 1)
  → FieldParser.parseFile() / reExtract() (Step 2)
  → 用户确认列映射、单位、多尺寸选择
  → confirmData() 生成 state.items
  → Step 3：用户调整 SKU 叠放/翻转、选择箱型（推荐或手动/混合）
  → startCalculation() → PackingEngine.calculate()
  → Step 4：结果摘要、箱标签、告警、3D 视图、PDF 导出
```

### 2.2 关键模块职责

| 模块 | 文件 | 职责 |
|---|---|---|
| App 控制器 | `app.js` | 4 步向导、状态管理、UI 渲染、模块协调 |
| 字段解析 | `field-parser.js` | Excel/CSV 解析、列语义映射、单位推断、多尺寸消歧 |
| 箱型数据库 | `container-db.js` | 6 种箱型参数、门约束、尺寸约束、重量约束、选箱推荐 |
| 装箱引擎 | `packing-engine.js` | EMS + 层铺混合算法、旋转、堆叠、稳定性、利用率计算 |
| 3D 视图 | `three-viewer.js` | Three.js 渲染装箱结果 |
| PDF 导出 | `pdf-exporter.js` | 结果报表导出 |

### 2.3 已发现的主要问题（来自上一轮审阅 + 本次探索）

#### A. 箱型推荐逻辑与规范不一致

1. **OT 类型拼写错误（已修复）**
   - 代码检查 `'opentop'`，数据库中为 `'openTop'`，导致 OT 门约束失效。
   - 修复后需回归验证。

2. **FR 推荐未考虑 6 方向旋转**
   - 原代码在推荐 FR 时只检查货物原始尺寸是否超出 `effectiveMaxDims`。
   - 规范要求：FR 无门约束，但货物必须能在 6 个旋转方向中放入有效尺寸。
   - 已通过新增 `fitsByRotation()` 修复。

3. **OT 门约束语义不清**
   - 规范：OT 只检查最小横截面 ≤ 门宽，高度不受门限制。
   - 代码：`doorH = Infinity`，标准柜分支中 `Math.max(effDoorW, effDoorH) = Infinity`，逻辑上实现了"高度不约束"，但可读性差，容易误解为"完全无约束"。

#### B. 尺寸/重量约束

4. **FR 有效尺寸计算**
   - `getEffectiveMaxDims()` 对 FR 使用 `L + maxOverLength`、`W + maxOverWidth`、`H + maxOverHeight`。
   - 规范中 FR 允许超宽/超高/超长，但这是否意味着可以三面同时超出？行业实践中通常有重心/安全限制，代码未体现。

5. **重量约束仅检查单件和累计**
   - 未检查重心分布（前后、左右重量平衡）。
   - 未检查单件重量是否超过叉车/吊装能力。

#### C. 堆叠与稳定性

6. **`orientationFixed` 语义与 UI 标签不一致**
   - UI 标签为"不可翻转"，代码中 `orientationFixed` 表示"只允许绕 Z 轴水平旋转（2 个方向）"。
   - 规范中应明确："不可翻转" = 不允许倒置/侧翻，但仍可水平旋转。

7. **非叠放货物处理**
   - `blockedAbove` 标志阻止在货物正上方放置其他货物。
   - 但 EMS 空间分割后，其他货物仍可能从侧面绕过并堆叠在非叠放货物旁边，这是否符合业务预期？

8. **`isStackable()` 函数未使用**
   - `container-db.js` 中导出 `isStackable()`，但 `packing-engine.js` 中直接读取 `item.stackable`。
   - 要么规范说明 `isStackable()` 是公共 API，要么清理死代码。

#### D. 字段解析与数据流

9. **重量字段统一映射为 `grossWeight`**
   - 规范说明：装箱算法仅需一个重量字段参与载荷计算，所有重量表头统一映射为 `grossWeight`。
   - 这可能导致业务误解：用户看到"净重"列被当作"毛重"使用。
   - 建议在 UI 中明确提示：若存在多列重量，优先使用毛重。

10. **`combinedDimension` 优先级高于独立 `length/width/height`**
    - 如果用户同时提供了组合尺寸列和独立尺寸列，代码优先使用组合尺寸。
    - 规范中应明确优先级，避免用户调整列映射后结果不变。

#### E. 性能与正确性 trade-off

11. **EMS 全量重建**
    - `layerPack` 每次层铺后都 `buildEMSFromPlaced(placedItems)`。
    - 规范应说明：当前算法以计算性能换取空间正确性，不保证最优解。

12. **`autoRetry` 未比较结果优劣**
    - 失败或 warning 时切换排序策略重新计算，但不比较两次结果。
    - 规范应明确：autoRetry 是"尽力而为"，不保证结果改进。

## 3. 建议的修复与改进计划

### 阶段一：规范对齐与回归验证（高优先级）

#### 3.1 修复 OT/FR 类型检查与推荐旋转
- **文件**：`container-db.js`
- **内容**：
  - 确认 `type === 'openTop'` 和 `type === 'flatRack'` 已正确区分大小写。
  - 确认 `fitsByRotation()` 在 FR 推荐中被调用。
- **验证**：运行 `dogfood-output/run-core-tests.js`，OT/FR 相关用例通过。

#### 3.2 明确 OT 门约束的代码注释与规范描述
- **文件**：`container-db.js`
- **内容**：
  - 在 `checkDoorConstraint()` 中补充注释：OT 只检查最小面 ≤ 门宽，高度不约束。
  - 将 `doorH: Infinity` 的处理逻辑显式化，避免依赖 `Math.max(...Infinity)` 的隐式行为。
- **验证**：代码审查 + 单测。

#### 3.3 统一 `orientationFixed` 的语义
- **文件**：`app.js`、`packing-engine.js`
- **内容**：
  - UI 标签"不可翻转"改为"不可倒置/侧翻（仅可水平旋转）"或类似更精确描述。
  - 在规范文档中明确：orientationFixed=true 时允许 2 个水平旋转方向。
- **验证**：检查 UI 文案和 `getOrientations()` 返回的方向数。

### 阶段二：规范文档编写（中优先级）

#### 3.4 编写《操作规范与代码映射》文档
- **文件**：`docs/container-loading-operation-spec.md`（或用户指定路径）
- **内容**：
  - 第一部分：通用操作规范（坐标系、旋转、门约束、尺寸、重量、堆叠）

  - 第三部分：选箱推荐逻辑
  - 第四部分：代码映射索引
  - 第五部分：常见异常与处理
- **验证**：文档与代码关键函数逐条核对。

### 阶段三：功能增强（低优先级，按需实施）

#### 3.5 重量约束增强
- **文件**：`container-db.js`、`packing-engine.js`
- **内容**：
  - 可选：增加重心分布检查（前后 60/40，左右 50/50）。
  - 可选：增加叉车单件重量上限提示。
- **验证**：新增单元测试。

#### 3.6 `autoRetry` 结果比较
- **文件**：`packing-engine.js`

- **验证**：模拟失败场景，确认取优逻辑。

- **文件**：`packing-engine.js`

- **验证**：对比优化前后的运行时间和装箱正确性。

## 4. 假设与决策

1. **规范权威性**：以当前代码已实现的约束为主，行业资料为辅。若代码与行业实践冲突，优先保证代码现有行为稳定，再逐步引入行业最佳实践。
2. **最小改动原则**：本次计划以文档化和对齐现有逻辑为主，不引入新的复杂功能（如重心计算）。
3. **测试策略**：继续以 `dogfood-output/run-core-tests.js` 作为回归基线，补充 OT/FR 用例。
